## Structure of an OpenAPI Specification

An <LinkOut href='https://spec.openapis.org/oas/latest.html'>OpenAPI</LinkOut> document typically contains the following sections:

### 1. **OpenAPI Version**
   - Specifies the OAS version used (e.g., `3.0.3` or `3.1.1`).
   - Required as it dictates the syntax and features available in the specification.

### 2. **Info Object**
   - Contains metadata about the API, such as:
     - `title`: The name of the API.
     - `description`: A brief description of the API.
     - `version`: The version of the API.
     - `termsOfService`, `contact`, and `license` are optional fields for legal and informational context.

### 3. **Servers Object**
   - Lists one or more server URIs where the API is hosted, along with optional descriptions.
   - Example: A `production` and a `staging` server.

### 4. **Paths Object**   
   - An **endpoint** represents a specific path (route) in the API, such as `/users` or `/products/{id}`. Each endpoint corresponds to a URI path and supports HTTP methods like **GET**, **POST**, **PUT**, **DELETE**, etc.   
   - Each path has operations (`GET`, `POST`, `PUT`, etc.) with details for each method:
     - `summary` and `description` of what the operation does.
     - `parameters` for inputs (path, query, header, and cookie parameters).
     - `responses` indicating status codes and corresponding response bodies.
     Example:
      ```yaml
        paths:
            /users:
            get:
                summary: Retrieves a list of users
                responses:
                200:
                    description: A list of users
        ```

### 5. **Components Object**
- The **components** object is a container for reusable definitions, such as **schemas**, **responses**, **parameters**, and **security schemes**. It promotes reusability and reduces redundancy across the API definition. It includes:
     - **Schemas**: Data models or definitions used in the API (e.g., JSON objects for request/response).
     - **Parameters**: Reusable parameters use`d across multiple paths.
     - **Responses**: Standardized response formats for reuse.
     - **Examples**: Sample data for responses and requests.
     - **RequestBodies**: Predefined request payloads.
     - **Headers**: Custom header definitions.
     - **SecuritySchemes**: Defines authentication methods (e.g., OAuth2, API keys).

### 6. **Security**
   - Specifies global or operation-level authentication and authorization.
   - Example: API keys, OAuth 2.0, and HTTP authentication (e.g., Bearer tokens).

### 7. **Tags**
   - Organizes operations into groups, making it easier to structure and navigate complex APIs.

### 8. **External Documentation**
   - Allows linking to additional documentation outside the OpenAPI document, providing supplementary context.

### 9. **Paths Parameters and Responses**
   - Details for each operation, including:
     - **Parameters**: Input parameters, such as query, path, header, and cookie parameters, with details on type, format, and constraints.
     - **Responses**: Expected HTTP status codes and their associated responses (headers, body content, etc.).
